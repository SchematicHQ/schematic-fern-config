The Manage Plan API lets you programmatically set a company's full plan state, including the base plan, add-ons, credit bundles, and pay-in-advance entitlements. It handles both Stripe-linked and non-billing plans in a single call.

<Warning>
This is a **replace-style (PUT-like) API**, not a patch. The request body represents the desired end state of the company's plan configuration. Any plans or add-ons not included in the request will be removed.
</Warning>

## Endpoints

| Endpoint | Purpose |
|----------|---------|
| `POST /manage-plan` | Apply a plan change |
| `POST /manage-plan/preview` | Preview the financial impact of a plan change without applying it |
| `POST /manage-plan/subscription/cancel` | Cancel a company's active subscription |

All endpoints require API key authentication.

## How it works

When you call `POST /manage-plan`, you provide the complete desired plan state for a company. Schematic then:

1. Determines which plans are billing-linked (connected to Stripe) and which are not
2. For billing-linked plans, updates the Stripe subscription accordingly
3. For non-billing plans, applies the assignment directly
4. Returns the updated company object

Because this is a replace-style API, you must always include the full desired state. For example, if a company currently has add-ons A and B, and you want to add C, you must include all three in `add_on_selections`. If you only send C, add-ons A and B will be removed.

### Plan version behavior

When specifying a base plan or add-on:

- If you provide an explicit `version_id`, that version is used
- If the company is already on that plan and you omit the version, the company's current version is preserved
- If it's a new plan and you omit the version, the latest published version is used

### Default (fallback) plan

If you clear all plans from a company (by omitting `base_plan_id`), Schematic will assign the configured default/fallback plan if one exists.

## Manage Plan

`POST /manage-plan`

### Request body

```json
{
  "company_id": "comp_abc123",
  "base_plan_id": "plan_xyz789",
  "base_plan_price_id": "bprice_def456",
  "base_plan_version_id": "planv_ghi012",
  "add_on_selections": [
    {
      "plan_id": "plan_addon1",
      "price_id": "bprice_jkl345",
      "version_id": "planv_mno678"
    }
  ],
  "credit_bundles": [
    {
      "bundle_id": "bundle_pqr901",
      "quantity": 5
    }
  ],
  "pay_in_advance_entitlements": [
    {
      "price_id": "bprice_stu234",
      "quantity": 10
    }
  ],
  "coupon_external_id": "DISCOUNT20",
  "promo_code": "LAUNCH2024",
  "payment_method_external_id": "pm_stripe_abc",
  "trial_end": "2024-12-31T23:59:59Z"
}
```

### Field reference

<ParamField path="company_id" type="string" required>
The ID of the company whose plan you want to manage.
</ParamField>

<ParamField path="base_plan_id" type="string">
The plan to assign as the company's base plan. Omit or set to `null` to clear the base plan (the company will fall back to the default plan if one is configured).
</ParamField>

<ParamField path="base_plan_price_id" type="string">
The specific price to use for the base plan. Required for billing-linked plans with multiple price options (e.g. monthly vs. yearly).
</ParamField>

<ParamField path="base_plan_version_id" type="string">
A specific version of the base plan. If omitted, the company's current version is preserved (same plan) or the latest published version is used (new plan).
</ParamField>

<ParamField path="add_on_selections" type="array">
The complete list of add-ons the company should have. Each entry requires a `plan_id`. Billing-linked add-ons also require a `price_id`. An optional `version_id` can pin a specific version.

Add-ons not included in this array will be removed from the company.
</ParamField>

<ParamField path="credit_bundles" type="array">
Credit bundles to include in the subscription. Each entry requires a `bundle_id` and `quantity`.
</ParamField>

<ParamField path="pay_in_advance_entitlements" type="array">
Pay-in-advance quantities (e.g. user seats). Each entry requires a `price_id` and `quantity`.
</ParamField>

<ParamField path="coupon_external_id" type="string">
A Stripe coupon ID to apply to the subscription.
</ParamField>

<ParamField path="promo_code" type="string">
A promotional code to apply.
</ParamField>

<ParamField path="payment_method_external_id" type="string">
A Stripe payment method ID to use for this subscription.
</ParamField>

<ParamField path="trial_end" type="datetime">
An ISO 8601 timestamp for when the trial should end. Must be in the future. Trials have restrictions: the plan must be marked as trialable, and you cannot combine trials with add-ons or usage-based entitlements.
</ParamField>

### Response

```json
{
  "data": {
    "company": { ... },
    "success": true
  }
}
```

The response includes the full updated company object reflecting the new plan state.

## Preview Manage Plan

`POST /manage-plan/preview`

Use this endpoint to preview the financial impact of a plan change before applying it. The request body is identical to `POST /manage-plan`.

### Response

```json
{
  "data": {
    "subscription_change_preview": {
      ...
    }
  }
}
```

The preview includes details like prorated charges, credits, and the new subscription amount. This is useful for showing users what a plan change will cost before they confirm.

## Cancel Subscription

`POST /manage-plan/subscription/cancel`

Cancels a company's active Stripe subscription. This only applies to companies with billing-linked plans.

### Request body

```json
{
  "company_id": "comp_abc123",
  "cancel_immediately": true,
  "prorate": true
}
```

<ParamField path="company_id" type="string" required>
The ID of the company whose subscription you want to cancel.
</ParamField>

<ParamField path="cancel_immediately" type="boolean">
If `false`, the subscription cancels at the end of the current billing period instead of immediately. Defaults to `true`.
</ParamField>

<ParamField path="prorate" type="boolean">
If `true` and `cancel_immediately` is `true`, a prorated credit is issued for the unused portion of the billing period. Defaults to `true`.
</ParamField>

### Response

Same format as `POST /manage-plan`, returning the updated company object.

## Common patterns

### Upgrading a plan

To upgrade a company from one plan to another, send the new plan ID. Schematic handles proration and Stripe subscription updates automatically.

```json
{
  "company_id": "comp_abc123",
  "base_plan_id": "plan_pro",
  "base_plan_price_id": "bprice_pro_monthly",
  "add_on_selections": []
}
```

<Note>
Sending an empty `add_on_selections` array will remove all existing add-ons. If you want to preserve them, include them in the array.
</Note>

### Adding an add-on while preserving existing state

To add an add-on without changing anything else, you must include the current base plan and all existing add-ons in the request.

```json
{
  "company_id": "comp_abc123",
  "base_plan_id": "plan_pro",
  "base_plan_price_id": "bprice_pro_monthly",
  "add_on_selections": [
    { "plan_id": "plan_existing_addon", "price_id": "bprice_existing" },
    { "plan_id": "plan_new_addon", "price_id": "bprice_new" }
  ]
}
```

### Removing all plans

Omit `base_plan_id` and send an empty `add_on_selections` array. If a default plan is configured, the company will be assigned to it. If the company has a Stripe subscription, it will be cancelled.

```json
{
  "company_id": "comp_abc123",
  "add_on_selections": []
}
```

## Error cases

- **Externally managed subscription**: If the company's subscription is managed outside of Schematic (e.g. directly in Stripe without the Schematic integration), the API will return an error.
- **Invalid trial configuration**: Trials require a trialable plan with no add-ons and no usage-based entitlements.
- **Missing price for billing-linked add-on**: Billing-linked add-ons require a `price_id`.
- **Price specified for non-billing add-on**: You cannot specify a `price_id` for an add-on that is not linked to a billing product.
